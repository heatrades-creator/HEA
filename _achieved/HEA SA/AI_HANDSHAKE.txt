WHAT THIS IS
--------------------------------------------------------------------------------

A fully-featured solar proposal and quoting platform built as a Google Apps
Script web application. It is a single-page application (SPA) deployed as a
Google Apps Script Web App, accessible via a public URL. No external hosting.
No paid services. Runs entirely on Google's infrastructure for free.

The platform is used by Jesse (the electrician) on a tablet or laptop during
in-home sales consultations. A customer's NEM12 electricity interval data
(downloaded from their retailer) is uploaded, the app analyses it against
solar generation models, sizes a system, runs a full discounted cashflow
analysis, generates a signed PDF quote, and emails it — all in one session.

--------------------------------------------------------------------------------
TECHNOLOGY STACK
--------------------------------------------------------------------------------

Runtime:         Google Apps Script (GAS) V8 engine
Frontend:        HTML/CSS/JavaScript served by HtmlService.createTemplateFromFile
Backend:         Google Apps Script .gs files (server-side JavaScript)
Storage:         Google Sheets (sessions, admin log)
File storage:    Google Drive (PDF quotes, job folders)
Email:           GmailApp.sendEmail()
Charts:          Chart.js 4.4.0 (CDN)
PDF generation:  HTML → Google Doc → export as PDF
Deployment:      Deploy → Web App → ANYONE_ANONYMOUS, executeAs USER_DEPLOYING
URL pattern:     https://script.google.com/macros/s/[DEPLOYMENT_ID]/exec

The app uses HtmlService template includes:
  Index.html includes: FinanceEngine.html, Quote.html, SecretBackend.html
  All .html files use <?!= include('filename') ?> server-side templating

IMPORTANT: FinanceEngine.html does NOT have <script> tags — it is injected
directly into the main <script> block in Index.html. Adding <script> tags to
FinanceEngine.html would cause a "Malformed HTML" crash.

--------------------------------------------------------------------------------
ARCHITECTURE DECISIONS (ACCEPTED — DO NOT RELITIGATE)
--------------------------------------------------------------------------------

1. ANYONE_ANONYMOUS access — intentional. Clients use this at sales meetings
   without logging in. Drive functions are PIN-gated separately.

2. ALLOWALL iframes — accepted risk. Needed for hea-group.com.au embedding.
   Documented in Code.gs with comment.

3. Two-engine split — client-side DCF (runEngine in FinanceEngine.html) for
   instant slider response; server-side builders (PDFBuilder) for PDF/email.
   These are not the same engine. The client engine outputs are serialised into
   engineSummary + engineProjection and passed to the server for PDF generation.

4. Geometric fallback — acceptable only as a fallback. doPDF() BLOCKS if
   _lastEngineResult is null. The geometric series only runs if engineProjection
   is absent (should not happen in normal use).

5. Optimiser uses approx payback for screening — 440 combinations × full DCF
   would freeze the browser. Tier 1 is fast approx screen. Tier 2 runs full DCF
   on top 10 shortlist. Labelled clearly in UI as "fast screen, not full DCF".

6. Public/admin split not done — the privileged functions (Drive, sessions,
   email) are in the same deployment as the public estimator. Drive is PIN-gated.
   Sessions are HMAC-signed. This is architecture debt, not a resolved concern.

SECURITY ARCHITECTURE
--------------------------------------------------------------------------------

Admin PIN gate:
  getDriveFiles(adminPin) and getDriveFile(id, adminPin) require the PIN.
  Wrong/missing PIN returns { error: 'auth' } or null.
  Logged to AdminLog sheet.
  Set HEA_ADMIN_PIN in Script Properties (throws if not set).

Session security:
  saveSessionForResume generates: UUID (Utilities.getUuid()) + HMAC-SHA256 sig
  loadSession(id, sig) verifies HMAC before touching spreadsheet.
  Set HEA_SESSION_KEY in Script Properties (throws if not set).
  Sessions expire after 90 days. purgeExpiredSessions() cleans up weekly.

Signature handling:
  Client draws signature on canvas → base64 data URI.
  Passed to server as data URI — NO Drive file created, NO public sharing.
  Embedded directly in PDF HTML.

HTML escaping:
  escHtml(s) exists in Config.gs.
  Applied at entry to buildReportHTML() and buildEmailHTML() for:
  clientName, postcode, quoteNo, result.nmi, result.zoneName.

FINANCE ENGINE — ACCOUNTING BASIS
--------------------------------------------------------------------------------

CUSTOMER CASHFLOW VIEW (not project/system view):
  Year 0: ICF0 = -cash_deposit (only what customer paid upfront)
  Annual: Annual_NCF_t = Savings_t - Maintenance_t - Replacements_t - Loan_Repayments_t
  (lean model: Fees=0, Insurance=0, Taxes=0, Incentives=0, Residual=0)

Four payback metrics:
  1. Approx Simple Payback = Upfront_Cost / Year_1_Net_Savings (shortcut only)
  2. CF+ Year = first year Annual_NCF_t > 0
  3. BEP = first year Cumulative_Nominal_NCF_t >= 0 (undiscounted)
  4. DPB = first year Cumulative_DCF_t >= 0 (discounted)
  Plus: NPV = ICF0 + SUM(DCF_t), IRR, MIRR

CRITICAL: CumCF_exact and CumDCF_exact store un-rounded values.
  Interpolation uses exact values. Display uses rounded CumCF/CumDCF.
  NEVER use rounded values for financial calculations.

CRITICAL: repayAmt, period, ppy, intRate are declared BEFORE the loanHint block
  in updateROI(). This was a real bug that caused the loan summary to show wrong
  values. Do not reorder these declarations.

Negative amortisation guard:
  checkLoanServicability() runs before runEngine().
  If repayment <= interest accrual, show error and stop.
  Do NOT model negative amortisation silently.

Loan amortisation:
  n_periods = -ln(1 - (Principal × r_period) / Repayment) / ln(1 + r_period)
  Bucketed into annual outflows for the DCF engine.
  Payment frequencies: weekly (ppy=52), fortnightly (ppy=26), monthly (ppy=12).

--------------------------------------------------------------------------------
NEM12 DATA FORMAT
--------------------------------------------------------------------------------

NEM12 is the Australian AEMO standard for electricity meter interval data.
Format: CSV with record types 100/200/300/500/900.
200 record: channel metadata (NMI, suffix/channel, UOM, interval length)
300 record: one day of 30-min interval data (48 values + quality flag)
Quality flags: A=actual, E=estimated, S=substituted

Parser validates:
  - parts.length >= 51 BEFORE parsing intervals (critical — prevents zero-fill)
  - isNaN check on each interval value
  - Date range: year 2000-2040, month 1-12, day 1-31
  - Exactly 48 numeric intervals required

Channel selection (import = load we can offset with solar):
  Accepted: E1, B1, IMPORT, NET_IMPORT (or empty suffix = single channel)
  Rejected: B2, E2, K1, Q1, NET_EXPORT, EXPORT (export/generation channels)
  Unknown: REJECTED by default, two-pass fallback warns loudly

Hard stops:
  <14 days of data → error
  >15% or >50 malformed rows → error
